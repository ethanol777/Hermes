## 故障根因分析

脚本中的 `set -e` 导致在 **Windows + core.autocrlf=true** 环境下的偶发崩溃：

1. `auto_sync.sh` 从 `~/.hermes/` 复制文件到 `~/Hermes/hermes/`（这些文件是 LF 换行符）
2. `git status --porcelain` 检测到 **M**（未暂存的修改），进入提交分支
3. `git add -A` 暂存文件，Git 触发 **LF→CRLF** 转换（stderr 出现 CRLF 警告）
4. 转换后，暂存区内容与 HEAD 完全相同 → `git commit` 退出码为 **1**（"nothing to commit, working tree clean"）
5. `set -e` 捕获到此退出码，立即终止整个脚本，返回 exit code 1

## 修复内容

在 `~/Hermes/hermes/scripts/auto_sync.sh` 第 78 行的 `git commit` 后添加 `|| true`，允许 "nothing to commit" 场景正常继续而不是崩溃。

```diff
-  git commit -m "auto-sync $(date '+%Y-%m-%d %H:%M')"
+  # Windows core.autocrlf=true 下 git add 转换 LF→CRLF 后内容可能无变更，
+  # 此时 git commit 返回 code 1（"nothing to commit"），用 || true 保护
+  git commit -m "auto-sync $(date '+%Y-%m-%d %H:%M')" || true
```

## 效果

下次 `auto_sync.sh` 运行时，若 `git commit` 因换行符转换导致无实际变更，脚本会继续执行 fetch/rebase/push 流程而非崩溃退出。
